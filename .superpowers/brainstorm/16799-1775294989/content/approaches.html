Three approaches to production-ready deployment

Pick the one that fits your goals

A

Minimal Fix

Fix the broken Dockerfile, resolve the port mismatch, bundle the PDF.js worker locally, add .dockerignore, and provide a Caddyfile snippet. Smallest scope — gets you to a working self-hosted container fast.

Pros

  • Fewest changes
  • Fast to ship
  • Easy to understand

Cons

  • Docker image still larger than needed
  • No deploy workflow
B

Production-Hardened ⭐ Recommended

Everything in A, plus: Next.js standalone output (cuts image size ~70%), fix the healthcheck, a .env.example for config, and a deploy.sh script for one-command updates on the VPS.

Pros

  • Lean image (~150MB vs ~500MB)
  • Simple deploy workflow
  • Proper env config pattern
  • Production-grade healthcheck

Cons

  • Slightly more files to change
C

Full DevOps Stack

Everything in B, plus: Watchtower for auto image updates, Makefile with make build, make deploy, make logs commands, and a GitHub Actions workflow for CI builds.

Pros

  • Full automation
  • CI/CD ready

Cons

  • Overkill for a single utility app
  • More moving parts to maintain
